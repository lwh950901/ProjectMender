# ProjectMender V1 设计规格

**日期：** 2026-07-16
**状态：** 待用户审阅

## 1. 目标与边界

ProjectMender 是桌面优先的本地代码项目治理工作台。用户选择一个本地目录后，应用分析代码质量、安全治理与技术债/工程化问题，给出按优先级排列的建议。用户选定建议后，应用以 OpenSpec 工件细化变更，生成可审阅补丁，并在用户确认后应用与验证。

V1 支持 macOS（Apple Silicon）与 Windows 11 x64。V1 不支持远程仓库导入、产品功能策划、自动执行命令、自动修改代码、原始敏感值出站或 Rust 专用深度规则。

## 2. 关键产品原则

- 本地扫描默认只读，不运行项目脚本、依赖安装、构建、测试、Git hook、Docker 或模型建议的命令。
- 代码、密钥和验证日志只有在用户明确确认的最小范围内才会发送到已配置的云端模型。
- AI 输出均为提案：OpenSpec 工件、命令和补丁都必须经过用户审核。
- 本地规则发现、依赖情报发现和 AI 建议分开标记，且每项都可追溯证据和覆盖状态。
- 未覆盖、无法扫描和扫描失败必须显式呈现，不能被表述为“无问题”。

## 3. 技术架构

采用 Tauri 2 桌面应用，React/TypeScript 负责界面和可复用业务模型，Rust 负责受控文件访问、钥匙串、进程、网络审批与补丁写入。

```text
用户
  -> Tauri UI
  -> 本地项目访问层
  -> 扫描编排器
  -> 建议清单
  -> 上下文出站网关
  -> 模型适配器
  -> OpenSpec 工件与状态机
  -> Diff 审核 / 受控验证
```

本地项目访问层只可访问用户选定项目目录和应用私有数据目录。它负责读取项目、记录 Git 指纹、写入经批准的 OpenSpec 工件和应用经批准的补丁。

## 4. 项目预检与扫描

### 4.1 项目识别

识别语言、包管理器、构建工具、依赖清单、Git 状态、CI 配置和 OpenSpec 状态。V1 支持 JavaScript/TypeScript、Python、Java、Go、HTML、CSS、Less、Sass 的项目识别；Rust 仅接受通用扫描。

### 4.2 OpenSpec 预检

扫描开始前执行只读预检：

1. 检测 `openspec/` 和配置文件。
2. 读取本地安装版本与项目内指引状态。
3. 在不写入的前提下运行 `openspec doctor`、`openspec validate --all --json` 和 `openspec status --json`。
4. 用户允许联网后，查询是否存在更新版本。

预检将“版本过期”“项目指引不同步”“现有规范/变更不健康”分开报告。应用不自动执行 `openspec init` 或 `openspec update`；两者均先预览写入内容并由用户确认。

### 4.3 扫描器分层

- 项目识别器：生成项目画像并决定可用适配器。
- 通用治理扫描器：敏感文件与泄漏模式、Git 忽略、许可证、依赖清单、生成物、超大文件、目录与配置问题。
- 语言适配器：针对已支持语言输出语法和工程规则发现。
- 依赖情报适配器：在单独网络授权后查询漏洞、许可和弃用信息；网络不可用时明确标为未检查。
- OpenSpec 适配器：执行预检、读取工件状态和在审批后管理工件。

所有扫描器必须输出统一发现结构：来源、类别、优先级、置信度、规则 ID、证据文件与行范围、影响、建议动作和覆盖状态。

### 4.4 发现方法与 AI 审查配方

ProjectMender 采用“两阶段发现”而不是让模型无约束地审阅整个仓库。

第一阶段是本地确定性发现，负责建立可复查的项目事实：

- Bug 与质量：语法/类型/lint 结果、复杂度、重复、异常处理、死代码、测试缺口、构建与配置问题。
- 安全治理：敏感文件和泄漏模式、危险 API/编码模式、依赖漏洞、许可证、Git 与 CI 安全配置。
- 技术债与工程化：依赖过期、模块循环或边界问题、目录与配置漂移、缺失 CI/测试/发布/可观测性能力、过大文件与生成物治理。

第二阶段是由用户从某条发现或明确模块范围触发的 AI 审查配方。配方是受版本管理的提示、输入契约和结构化输出要求，不是安装到用户项目中的 Superpowers 技能。V1 提供：

1. **Bug 核实**：根据规则证据、最小代码上下文和可选日志判断是否成立；输出复现条件、根因、影响范围、最小修复和测试建议。
2. **安全治理复核**：输出攻击前提、证据、严重性、误报判断、修复优先级与安全测试建议；不生成或执行攻击载荷。
3. **重构与技术债设计**：输出当前边界、目标边界、迁移步骤、兼容性/回滚风险、受影响测试和 OpenSpec 变更范围。
4. **工程化改进**：输出 CI、测试、依赖、构建、发布或可观测性缺口的最小改进方案与验证命令候选。
5. **验证失败返工**：仅使用已批准的失败日志、当前 diff、规则证据和最小相关代码，输出失败原因假设、所需额外证据与下一轮最小补丁建议。

每个配方必须声明：允许的输入类型、最大上下文预算、敏感内容排除规则、结构化输出 schema、可产生的 OpenSpec 工件和禁止的行为。配方不得自行扩大文件范围、调用项目命令、写入源码或绕过出站网关。

### 4.5 诊断流水线与动态过程展示

“发现问题”是 V1 的核心体验。扫描界面不得只显示无解释的进度条或最终报告，而应把诊断过程作为一条可回看的实时流水线呈现。它的目的不是暴露实现噪声，而是让用户理解：应用当前在检查什么、为何检查、已取得何种证据、哪些部分没有覆盖，以及结论来自本地规则还是 AI 复核。

默认展示为一条按阶段推进的诊断时间线：

```text
项目接入
  -> 项目画像
  -> OpenSpec 预检
  -> 本地规则与结构扫描
  -> 依赖情报（若获联网授权）
  -> 证据关联、去重与风险排序
  -> 发现清单
  -> （用户选中后）最小上下文打包与 AI 复核
```

每个阶段都是独立、可增量更新的任务单元，具有 `pending`、`running`、`completed`、`skipped`、`failed` 五种状态，并实时呈现：

- **正在做什么**：人可读名称、目的、当前适配器/规则组、已耗时和可取消操作；例如“分析 TypeScript 导入关系，用于发现循环依赖”。
- **发现与证据摘要**：目前发现数量按严重度和类别滚动更新；完成后显示规则数量、涉及文件数、主要问题簇及每项证据入口。
- **覆盖与跳过原因**：扫描文件数/候选文件数、未支持语言、排除路径、二进制/生成物/超大文件、无网络授权、工具不可用或扫描失败都必须显式列出，不能折算为“无问题”。
- **安全边界状态**：本地处理、已脱敏待确认、已阻断敏感内容、等待用户网络确认等状态清晰区分；扫描本身不把代码发送给模型。
- **下一步依据**：阶段结束时说明为何进入下阶段，例如“已收集 28 条原始信号，正在按共同模块和根因合并为问题簇”。

动态展示采用“概览优先、按需下钻”：顶部显示阶段时间线、总体覆盖率、发现计数和当前操作；中部展示新出现或已归并的问题簇；用户展开阶段后才看到适配器版本、规则 ID、文件清单、耗时、错误输出摘要和排除理由。原始文件内容、原始密钥值与完整模型上下文不在流水线中直接展示。

结果不是等全部扫描结束才一次性出现。确定性扫描器每产生一个可用发现即写入统一发现流；关联器随后以“已归并/优先级已调整”的可见事件更新它。扫描器失败不阻塞无关阶段：时间线标记失败范围，保留已完成发现，并提供“重试该阶段”入口。用户取消时保留已完成阶段和不完整覆盖说明，不生成“扫描通过”的结论。

AI 复核是独立的后置阶段，只有用户选择某个问题簇或范围后才出现。它显示将要打包的最小上下文、脱敏/排除项、token/成本估算与确认状态；确认前状态为“待发送确认”，并不代表任何代码已经出站。模型返回后，时间线把“AI 观点”与本地证据并列展示，且允许用户看到其引用了哪些已批准输入。

### 4.6 发现引擎：统一事件、证据图谱与问题簇

动态流水线的事实来源是发现引擎的统一事件流，而不是 UI 对各扫描器输出的猜测。所有适配器只能追加以下三类事件：

- **阶段事件**：阶段开始、进度、完成、跳过、失败或取消；包含输入范围、已处理/候选数、耗时、原因与覆盖影响。
- **证据事件**：某条规则或解析结果产生的不可变事实；包含稳定规则 ID、规则版本、文件与行范围、结构化摘要、置信度、类别和敏感性标记。证据不会被 AI 改写。
- **关联事件**：将证据加入、移出或重新计算某个问题簇；包含关联理由、优先级变化原因及受到影响的发现数。

扫描器可并行运行，但事件按单次扫描会话的递增序号写入本地事件账本，使界面、报告和后续 OpenSpec 工件能复现同一过程。用户重试一个阶段时生成新的阶段实例，不覆盖旧证据；过期证据以项目快照/文件内容指纹标记，不能与当前文件混淆。

问题簇是用户操作和 AI 审查的最小单位，而不是单条工具告警。关联器优先使用确定性关系：同一规则和同一代码位置、同一依赖及其传递路径、同一配置键、同一模块/调用或导入边界。只有具备明确依据的发现可以合并；不确定时保留为独立发现并显示“尚未关联”，避免为了减少数量而掩盖不同根因。

每个问题簇必须显示“为什么被排在这里”。优先级由可解释的四个维度构成：

```text
优先级 = 潜在影响 × 证据可信度 × 波及范围 × 修复价值
```

- **潜在影响**：安全/正确性/可用性/治理后果和严重度。
- **证据可信度**：解析或精确规则命中高于启发式信号；AI 复核只能增加说明与置信度，不能取代本地证据来源。
- **波及范围**：入口可达性、调用者/依赖者数量、生产配置或 CI 影响范围。
- **修复价值**：风险降低、可验证性和估算改动面；它不等同于“修起来容易”。

该公式产出高/中/低优先级和可解释的维度说明，而非伪精确的单一分数。安全严重度有最低优先级下限：已确认的敏感信息泄漏或高危生产暴露不能因改动成本高而被降至低优先级。用户可按类别、证据来源、可信度、覆盖状态或模块筛选，但不能修改原始证据。

### 4.7 规则包、扫描计划与质量门槛

发现能力以随应用发布的、版本化规则包提供，不依赖目标项目预先安装 lint、测试或构建工具。规则包分为：跨语言通用治理规则、语言/框架适配规则、依赖与许可证规则、OpenSpec/工程配置规则。每条规则有稳定 ID、版本、适用语言/文件条件、严重度建议、证据类型、已知局限和最小修复建议模板。

项目画像完成后，编排器生成并先显示“扫描计划”：将运行哪些规则包、预期检查的文件范围、需要的本地解析器、可选网络阶段、明确排除项和预计资源预算。用户可在运行前关闭某一规则组，但不能借此把未运行项目显示为通过；报告将其标记为“用户跳过”。规则包更新只随应用更新交付，V1 不提供从网络下载任意第三方规则或执行任意扫描脚本的能力。

规则按证据能力分级：

- **精确规则**：语法树、依赖清单、锁文件、Git/CI 配置等可直接验证的事实；可给出较高置信度。
- **关联规则**：基于导入图、调用关系、重复片段或多文件配置的确定性推导；必须列明关联链路。
- **启发式规则**：复杂度、命名、测试覆盖迹象、文件体量或模式组合；默认中低置信度，措辞为“需要检查”，不能直接声称存在 bug 或漏洞。

每条规则在进入发布规则包前必须具有正例、反例、边界例和跨平台 fixture；记录误报基线与预计耗时。出现解析失败、超时或不支持的语法版本时，该规则的覆盖范围降级并写入事件流，不允许用空结果替代失败。规则引擎为每个项目设定文件数、单文件大小、解析时间与总扫描时间预算；达到预算时优先完成安全和项目入口相关文件，并明确列出未扫描范围及可由用户触发的“继续深扫”操作。

用户反馈“误报/不适用”不会删除原始证据，而是为当前项目和规则版本创建本地抑制记录，必须填写理由并可随时撤销。抑制只影响排序与默认展示，不会影响敏感文件阻断或原始扫描覆盖报告；规则版本更新后需要重新确认是否沿用。

### 4.8 本地基线、增量扫描与结论变化

项目治理是重复发生的过程，V1 必须能区分“本次新引入的问题”与“此前已知问题”，但不把用户项目同步到云端。每个完成的扫描会话可由用户设为本地基线；应用仅在私有数据目录保存项目匿名标识、项目快照/提交标识、规则包版本、文件内容指纹、问题簇 ID、证据摘要、优先级和覆盖元数据，不保存源代码副本、原始敏感值或模型上下文。

后续扫描以最新有效基线比较，并将问题簇显示为：

- **新增**：当前会话出现、基线中没有等价证据的问题。
- **持续存在**：证据或同根因问题仍存在；文件移动/重命名但内容指纹可匹配时保持关联。
- **已解决候选**：基线问题在当前已覆盖范围内未再出现；若相关文件未覆盖、规则被跳过或解析失败，只能显示“待复核”，不能宣称已解决。
- **结论变化**：规则版本、项目快照、证据或优先级发生改变；必须说明是代码变化、规则变化还是覆盖变化。

“等价”优先由规则 ID、结构位置、文件内容指纹和确定性关联链路判断；不能可靠匹配时宁可显示为新增与旧问题待复核，也不进行猜测性合并。用户可选择将本次扫描设为新基线、保留历史基线或删除某项目的本地历史；删除只移除 ProjectMender 私有数据，不修改项目目录和 Git 历史。

增量扫描可先根据 Git 变更与文件指纹优先处理受影响文件及其导入/依赖邻域，但不能把它视为完整扫描。界面明确标注“增量覆盖”，并提供完整扫描入口。安全规则、项目根配置、依赖清单和锁文件在增量扫描中始终重新检查，因为少量变更可能影响整个项目。

### 4.9 受控扫描 Worker 与第三方能力接入

V1 不调用目标项目的 lint、构建、测试、包管理或任意脚本来发现问题：它们可能未安装、带有副作用、依赖网络或执行不受信任代码。ProjectMender 使用应用自有、版本锁定的扫描 worker；worker 只接收已获授权的项目范围，输出受限的结构化结果，且永不继承模型凭据。

扫描后端分为三层：

1. **内置读取与解析层**：目录枚举、忽略策略、文件分类、清单/锁文件/CI 配置解析、Git 元数据读取、轻量语法树和导入图；这是项目画像、覆盖统计和通用治理规则的可信基础。
2. **内置规则运行层**：运行 ProjectMender 版本化规则包，负责通用文本/结构匹配、复杂度、重复、模块关联和统一证据输出。
3. **受控外部扫描 worker**：仅在随应用打包、版本固定、许可证审查和跨平台验收后启用；其 stdout/stderr/报告先在本地脱敏和规范化，再转换为证据事件。worker 的临时文件只能写入应用私有临时目录，退出后清理。

V1 对外部后端采用“能力契约”而非把产品逻辑绑定到某个 CLI：秘密检测后端必须支持目录扫描、大小/超时限制、机器可读报告和输出脱敏；结构规则后端必须支持 JavaScript/TypeScript、Python、Java、Go、HTML/CSS 相关文件的离线规则执行；依赖后端必须能解析常见清单/锁文件并区分本地解析与联网漏洞查询。其候选实现将在技术 Spike 中按 macOS Apple Silicon、Windows x64、离线行为、许可证、输出是否含敏感值、资源占用和稳定 JSON/SARIF 输出逐项验收后确定。

当前候选包括语义/结构规则运行器、秘密检测器和 OSV 依赖漏洞扫描器。以 Gitleaks 为例，它支持目录和 Git 仓库扫描、JSON/SARIF 报告、大小/超时限制与输出脱敏，但其上游已宣布功能完成、后续主要接收安全修复；因此它只能作为 Spike 候选，不应被视为长期规则能力的唯一依赖。[Gitleaks 文档](https://github.com/gitleaks/gitleaks) 。OSV-Scanner 适合作为依赖漏洞信息候选，但其联网查询仍必须走独立的依赖情报授权。[OSV-Scanner 文档](https://google.github.io/osv-scanner/) 。

外部 worker 的安全边界需通过独立 P0 Spike 验证：应用只以非 shell 方式启动其固定可执行文件，为它建立无 API Key 的最小环境；只提供位于应用私有临时目录的、经路径策略筛选的只读扫描副本；并验证 macOS 与 Windows 上对项目原目录写入、任意子进程和网络访问的阻断或可审计控制。普通子进程会继承用户权限，不能把“直接启动 CLI”误称为沙箱。若上述跨平台控制无法可靠证明，外部 worker 不进入 V1，改由内置解析/规则层覆盖，或在后续具有专门沙箱能力的版本引入。

每次 worker 调用都产生可展示的执行清单：二进制/规则版本、输入文件数、隔离状态、网络状态、超时与大小上限、退出状态和报告规范化结果。若 worker 不可用或隔离验证不通过，应用保留其他层结果，并在流水线中显示“该能力未覆盖”；绝不退化为运行项目自身命令或上传整个目录给模型。

## 5. 建议清单与状态

结果按以下三组呈现，组内按优先级排序：

1. 代码质量：复杂度、重复、错误处理、测试、类型和风格。
2. 安全治理：敏感信息、危险编码模式、依赖风险、许可证和安全配置。
3. 技术债与工程化：依赖更新、构建/测试/CI 缺口、目录边界、重复配置和重构机会。

建议状态为：

```text
发现 -> 已选中 -> 待发送确认 -> 提案待审 -> 实施待审
-> 补丁待审 -> 已应用 -> 验证中 -> 已验证 / 失败分析待审 / 已忽略
```

验证失败时，应用创建失败分析包，用户可审核后进入“修复补丁 -> Diff 审核 -> 再验证”循环。默认连续失败三次后暂停，用户选择缩小范围、恢复某轮补丁、继续或结束。

## 6. 敏感数据与模型出站网关

所有模型请求均必须通过同一个出站网关。

### 6.1 默认规则

- `.env`、私钥、凭据目录、令牌、数据库密码、连接串等敏感内容先由本地规则分析，默认不发送。
- 命中内容使用 `[REDACTED]` 替换；只在必要时保留配置键名、服务类型或结构。
- 原始私钥、访问令牌、密码和其他敏感值在 V1 永远不可解锁或发送。
- 符号链接、嵌套仓库、二进制、归档、生成物、日志、数据库导出和超大文件都必须有明确的扫描与出站策略。

### 6.2 按次确认

每次模型调用前展示：模型供应商与模型名、目的、精确文件和代码片段、脱敏/排除项目、token/成本估算及预算裁剪。确认仅对该请求有效。用户可取消或缩小范围。

依赖情报联网与模型调用是两条独立的网络授权路径。

## 7. 模型与凭据

设置页面只包含供应商、模型名和 API Key。API Key 只保存于系统钥匙串，永不写入项目目录、报告、OpenSpec 工件或审计日志。

V1 先支持 DeepSeek 和 OpenAI-compatible API。模型接口必须支持流式输出、用户取消、超时、限流和鉴权错误处理。

`@earendil-works/pi-ai` 作为技术 Spike：验证其 DeepSeek/OpenAI-compatible provider、流式响应、token/成本统计、Windows/macOS 打包和系统钥匙串桥接。若采用，它必须运行在没有项目目录权限的 Node sidecar 中，且只接收已获批准的上下文与临时凭据。不得使用其默认文件凭据、环境变量解析、工具执行或 `providers/all`。

`@earendil-works/pi-agent-core`、`pi-coding-agent` 与 `pi-tui` 不进入 V1，以避免与 ProjectMender 的人工审批状态机和 Tauri 安全边界冲突。

## 8. OpenSpec 工作流

OpenSpec 是 V1 唯一正式变更工件与状态来源，Superpowers 不内置到用户项目中。

```text
扫描发现
  -> 用户选择建议
  -> AI 起草 proposal
  -> 用户审批 proposal
  -> AI 生成 design、spec delta、tasks
  -> 用户审批实施范围
  -> AI 生成补丁
  -> 用户审核 Diff
  -> 应用补丁
  -> （可选）批准验证命令
  -> 验证或返工循环
```

Superpowers 可作为未来外部编码代理 handoff 的兼容目标：ProjectMender 可导出已批准的 OpenSpec change、验证命令和安全边界，但不向目标项目强制写入其技能或 Agent 配置。

## 9. Git、补丁与命令执行

补丁生成时记录 Git 状态和目标文件内容指纹。

- 工作区有未提交改动时仍可扫描与生成建议，但应用补丁时必须提示冲突风险。
- 目标文件在生成后变化时，旧补丁不可直接应用。
- 用户可仅导出 patch；源代码写入必须经过 Diff 审核。
- 默认只读。用户可将当前项目标记为可信，随后执行已识别的常规验证命令；首次执行或命令变化仍需展示完整命令、工作目录和目的并确认。
- 模型建议命令无论项目是否可信都必须逐次确认。
- 每次执行保存退出码、输出摘要和改动文件列表。

## 10. 复用项目策略

- `alibaba/open-code-review`：借鉴确定性选文件、规则匹配、预算、预览和可恢复会话；先做适配器 PoC，不把其默认出站行为作为安全边界。
- `DeusData/codebase-memory-mcp` 与 `CodeBendKit/codeseek`：针对大型项目上下文检索二选一 PoC。先验证前者的成熟度与索引成本；若过重，改用后者。
- `rajudandigam/agent-inspect`：可选开发期观测工具，不作为产品运行依赖。
- `cobusgreyling/loop-engineering` 与 `getagentseal/codeburn`：仅借鉴循环审计、成本账本、钥匙串隔离和失败恢复设计。
- `TencentCloud/CubeSandbox`：后续 Linux/KVM 隔离验证能力；不进入 Windows/macOS 桌面 V1。

## 11. 故障、审计与隐私

本地审计只记录时间、项目标识、工具/规则版本、模型供应商/模型名、批准决定、补丁 ID 和验证状态，不保存 API Key 或原始敏感值。

- 扫描器失败：其他扫描继续，失败范围标为未覆盖。
- 模型失败：不改变项目，保留草稿与审批状态，可重试或切换模型。
- 出站检查失败：默认阻止发送。
- 补丁冲突：停止并要求重新生成或人工处理。
- OpenSpec 预检异常：报告异常；不擅自修复、更新或初始化。

报告导出默认不包含敏感内容，并在导出前提示其可能含项目路径和安全发现。

## 12. V1 验收标准

- 在 macOS Apple Silicon 与 Windows 11 x64 完成选择本地目录、只读扫描、结果分类、敏感内容阻断、模型发送预览、OpenSpec 提案、Diff 审核、补丁应用与验证返工闭环。
- 对支持的语言正确识别基础项目信息；对未支持/失败部分明确显示覆盖不足。
- 未获确认的上下文绝不发送；无论用户如何扩大范围，原始敏感值绝不发送。
- 未提交 Git 改动不会被静默覆盖；文件变化后的旧补丁不可应用。
- 任何命令都在运行前展示并等待确认。
- 扫描期间以动态流水线展示阶段、当前操作、发现摘要、覆盖范围、跳过/失败原因和安全边界状态；取消或局部失败时，保留已完成结果并明确标注不完整覆盖，绝不显示为“扫描通过”。
- 每条问题簇可追溯至不可变证据和关联理由；重复扫描、局部重试、并行完成顺序变化或文件变更不会静默篡改既有扫描会话的结论。
- 规则包具有版本、适用范围和正反例 fixture；跳过、超时、解析失败或预算耗尽均以未覆盖范围呈现，不能被空结果或低风险结论替代。
- 本地基线能正确区分新增、持续存在、已解决候选、待复核和规则/覆盖导致的结论变化；删除历史不触及项目文件，且历史数据不含源代码副本或敏感值。
- 扫描不运行目标项目命令。任何随应用打包的第三方 worker 均具有固定版本、受限输入/临时目录、超时与机器可读脱敏输出，并通过双平台隔离 Spike；不可用或隔离不成立时明确报告能力未覆盖。

测试包括：敏感检测/脱敏、预算、状态机、Git 指纹和补丁冲突的单元测试；多语言与 OpenSpec fixture 的集成测试；双平台的端到端模拟模型测试；以及符号链接、编码变体、超大文件、嵌套仓库、提示注入和模型恶意命令的人工安全测试。

验收还应覆盖每种 AI 审查配方：只接收契约允许的输入；输出满足结构化 schema；不能扩大上下文或触发命令；并能将建议正确映射到 proposal、design、spec delta、tasks、补丁或验证失败返工状态。
