## Context

ProjectMender 从空仓库开始，后续阶段将读取用户本地项目、调用云端模型、生成 OpenSpec 工件并可能运行验证命令。这些能力不能直接交给 UI 或任意插件；阶段 1 先建立双平台桌面壳、受限目录访问、钥匙串凭据和可扩展的授权状态，而不引入扫描、网络、进程或项目写入。

## Goals / Non-Goals

**Goals:**

- 在 macOS Apple Silicon 与 Windows 11 x64 上运行同一套 Tauri 2 桌面应用。
- 使 Rust 核心成为项目文件、应用私有数据和凭据的唯一受控入口；React UI 只通过窄命令接口请求操作。
- 对每次项目选择建立可验证的根目录能力，并阻止越出根目录的路径、符号链接和相对路径逃逸。
- 用操作状态模型表达将来的确认边界，同时保证阶段 1 没有可用的网络、子进程或项目写入路径。

**Non-Goals:**

- 不识别、解析或扫描项目内容，不运行 OpenSpec CLI，不调用模型，不执行任何项目命令。
- 不实现补丁、Diff、报告导出、后台自动更新、遥测、账号体系、远程仓库或 VS Code 集成。
- 不在此阶段接入第三方扫描 worker、agent loop 或模型 provider。

## Decisions

### 1. Tauri 2 采用“React UI + Rust 能力核心”分层

React/TypeScript 负责页面状态、目录选择后的展示和确认界面；Rust 命令层负责所有文件系统、应用数据、钥匙串和授权状态持久化。UI 不持有可重用的绝对项目路径权限，也不直接调用 Node、shell 或网络 API。

选择 Tauri 而非 Electron，是为了使用原生 WebView 与 Rust 的窄能力接口，降低打包体积和默认权限面。纯 Rust UI 会降低前端复用和迭代速度；Electron 则需要在主进程/预加载层重新建立相近的隔离边界。

### 2. 项目访问使用会话能力令牌，不以任意路径参数作为授权

用户在原生目录选择器中选择目录后，Rust 层规范化其根路径、生成随机 `projectSessionId` 并将根路径仅保存在本地会话/私有存储。后续 UI 命令使用 `projectSessionId` 加相对路径请求；Rust 层拒绝绝对路径、`..` 穿越、根目录外的规范化结果和指向根目录外的符号链接。

读取 API 先于写入 API 建立。阶段 1 只实现受限元数据读取和路径策略验证；任何写入入口、进程启动与网络客户端均不注册。相比“UI 传目录路径、Rust 每次检查”的方案，能力令牌更能避免 UI 或未来模型输出扩展访问范围。

### 3. 应用私有数据、项目数据与凭据三类存储物理分离

扫描会话、审批元数据和未来基线保存在应用私有数据目录，使用版本化本地数据模型；项目目录只在后续获得单独批准后才允许写入 OpenSpec 工件或补丁。API Key 仅通过 OS 凭据存储适配器读写，普通配置、数据库、日志、错误报告与序列化状态均只允许保存 provider、模型名和钥匙串引用标识。

选择“钥匙串适配接口”而不把具体库泄露给业务模块，使 macOS Keychain 和 Windows Credential Manager 的差异被隔离。文件式加密密钥库被拒绝为默认方案，因为它仍需处理主密钥位置、迁移和意外导出风险。

### 4. 授权状态先建模，危险执行后接入

授权记录至少包含操作类别、项目会话、范围摘要、状态（`pending`、`approved`、`rejected`、`cancelled`、`expired`）、创建/失效时间与不可逆审计 ID。阶段 1 的类别预留 `network_dependency_lookup`、`model_context_send`、`openspec_write`、`patch_apply`、`validation_command`，但所有执行适配器均不存在。

批准令牌默认单次、范围绑定、可过期；“项目可信”也不能在阶段 1 或后续自动批准模型出站、模型建议命令、补丁写入或密钥访问。这样后续阶段添加执行器时只能消费明确授权，而不会把 UI 提示语当作权限。

### 5. 本地审计仅记元数据并使用可恢复错误模型

基础审计记录操作类型、时间、项目会话匿名 ID、结果和错误分类，不记录文件内容、绝对路径、API Key 或原始敏感值。前端将错误显示为可操作的类别（目录不可访问、权限失效、钥匙串不可用、数据迁移失败），而非展示底层凭据或文件内容。

本地数据模型从第一版开始带 schema version 和迁移入口；迁移失败时以只读恢复模式启动，并允许用户清除应用私有数据，不修改项目文件。

## Risks / Trade-offs

- [符号链接或 TOCTOU 变化导致访问越界] → 每次受限读取重新验证根目录与目标路径；阶段 1 不提供写入，并在后续写入阶段增加文件指纹与二次确认。
- [不同平台的钥匙串行为、锁定状态或权限提示不同] → 以适配器接口和双平台集成测试覆盖保存、读取、删除、锁定与不可用错误。
- [未来子进程继承用户权限] → 阶段 1 不注册进程执行器；外部 worker 仅在独立隔离 Spike 通过后才进入后续阶段。
- [本地数据损坏或 schema 变化] → 版本化迁移、恢复模式和仅删除应用私有数据的重置选项。
- [Tauri 插件或打包差异扩大攻击面] → 最小化插件和命令白名单；每新增权限在对应 OpenSpec change 中声明、测试并审核。

## Migration Plan

这是空仓库的首次交付，没有旧版用户数据或项目格式需要迁移。应用首次启动创建其私有数据根和初始 schema；创建失败不得回退到项目目录。将来 schema 变更采用前向迁移与可恢复失败标记，卸载或“删除全部本地历史”只清除应用私有数据及用户选择移除的钥匙串条目。

## Open Questions

- 双平台签名、公证、安装包分发和自动更新策略在发布准备阶段确定，不阻塞阶段 1 本地开发与测试。
- 钥匙串条目的最终 service/account 命名规则将在具体凭据适配器 Spike 中确定，但必须支持按 provider/模型安全定位与用户删除。
